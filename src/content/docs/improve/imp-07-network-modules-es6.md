# 网络、模块、ES6+ 与常见设计模式

## 网络层：`ajax` / `axios` / `fetch` 心智

### 传统 `XMLHttpRequest` / `$.ajax` 系

- **历史包袱**：`XHR` API 偏底层，回调与事件模型**不够现代**；链式/组合不如 **Promise 优先** 的工具链好维护。
- **MVC/回调地狱**等属于**时代语境**；在旧项目**维护**时仍会碰到。

### `axios`

- **基于 `XHR`** 在浏览器、**基于 `http`** 在 Node；**统一**以 **Promise** 为默认异步模型。
- **拦截器、超时、**数据**自动** JSON、**可取消**（随版本/适配器**变化**需查文档）等工程能力完善。

### `fetch`

- **浏览器原生**、**Promise** 化；`fetch(url)` 在**仅网络错误/连接失败**时**reject**；**HTTP 4xx/5xx 默认仍 `resolve` 到** `Response`（需**手动**看 `ok` 或**抛错**再 `catch`）。
- **默认同源 cookie** 策略、**CORS 预检** 行为与 `XHR` **不完全相同**；`credentials`、**流式** `body`、**流控** 与**进度**能力依版本/环境差异大。

> 新代码在需要**统一**错误/取消/重试/拦截时，很多团队**仍**选 **axios/ky/封装层**；`fetch` 适合**更薄**的封装和**可渐进增强**。

## CommonJS 的**模块缓存**

`require()` **首次**执行模块代码，**结果**缓存在 `require.cache`（以**解析后的**绝对路径**为**键，概念如此）；**后续**同路径 `require` **直接**返回**同一份** `module.exports` **引用** → **多文件共享状态**的**单例**直觉、**循环依赖**的**部分导出**、**热更新**的 `delete` 等技巧都与此有关。

> **不要**在业务中随便 `delete require.cache` 做「黑魔法热更」，**框架**与 **打包器** 才是正道。

## ES6+ 与每日清单（可自查）

- 解构、扩展、**模板字符串**、**箭头**、`class` **语法糖**、`Map` / `Set`、**`Proxy`/`Reflect`**、**`Promise`/`async`** 等
- 细节对比见下：`let/const` vs `var`、**块级作用域/暂时性死区**、**箭头与普通函数**的 `this/arguments/构造能力` 等

## `let` / `const` / `var`（简表）

- **块级作用域** vs **函数量级/顶层** 的**漏到全局**（`var`）
- **提升**与**暂时性死区**（`let/const` **不会**在声明前**可用**）
- **重复声明**（`const/let` **不可**在**同层**重复同标识符，规则细查规范/ lint）
- **全局**对象上的**属性**污染（`var` vs `let/const` 在**全局**环境的行为差异在**浏览器/Node 模式**有差别，**以环境为准**）

## 箭头函数 vs 普通函数

- 箭头**无** `prototype`、**不能**作**构造函数**、**无** 自有 `this/arguments`（`arguments` 在箭头里**继承外层**的**非箭头**的 `arguments` 行为视引擎与严格模式、是否 rest 而变；**最稳**的仍是 **`...args`**）。
- **`call/apply/bind`** 对**箭头**的 `this` **无效**（**词法**继承）。

## `Map` 与 `WeakMap`

- **`Map` 的键**可为**任意**（含**对象**），按**强引用**保留键值；遍历顺序是**插入序**（ES 规范有精确定义）。
- **WeakMap** 的键**必须是 object**（以 ECMA-262/引擎为准，勿与 `null` 混淆作键），值侧为**弱**引用**键对象**所关联的数据；**不可**做「遍历全部 key」的集合用法；常用 API 如 `get` / `set` / `has` / `delete`。

> **WeakMap 典型用法**：`DOM 节点` ↔ 元数据 的**侧表**、私有字段 polyfill 思路、避免**全局**大 Map **钉死**已不需要的**对象**。

## **装饰器**（ES 提案/TS 实现）

- **类/方法/字段**上的**语法糖** → 多转译为**高阶函数**、**对目标做包装**；**目前**在 **TS** + **Babel/编译器** 中常见，**标准**仍在**演进**中（具体**阶段**、**可装饰位置** 以**你的编译链版本**为准）。
- 普通**函数声明**的**提升**、**可装饰时机** 与**类**不同，是「为何常见装饰器在**类**上**先**落地」的原因之一。

## ES Module 与 CommonJS

| 维度 | **ESM** | **CJS** |
| --- | --- | --- |
| 语法 | `import` / `export` | `require` / `module.exports` |
| 加载 | **可**在规范与打包器上实现**静**态分析/摇树/异步边界 | **多**在 Node 传统里**同步** `require`（`import()` **动态** 属另一话题） |
| 互操 | 打包与 Node 互操作需留意 `default`、\`__esModule\` 标记、循环依赖等 | 大量存量包仍是 CJS |

> Vite/rollup/webpack 的**预打包**与 Node 的 **「条件导出/exports 字段」** 把现实弄复杂了，**以工程配置为准**。

## 常见**设计模式**（名词表）

- **单例**、**工厂**、**策略**、**装饰器/包装**、**观察者**、**发布-订阅**（**通道** 与**直接**引用不同）  
- 在 JS/TS 里，大量模式被**高阶函数**、**类** 与**事件/流** 组合**替代**；**理解意图** 比 背 23 个名字 更重要。
