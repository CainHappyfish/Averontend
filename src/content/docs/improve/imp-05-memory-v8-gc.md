# 内存泄漏与 V8 / 垃圾回收

## 什么是内存泄漏

**内存泄漏**：程序在**不再需要**某块内存时，**仍**因**可到达**的**引用链**而**无法**被**回收**；并非物理内存消失。典型后果：**越跑越大**、**卡顿**、**溢出**。

## 常见根因

### 1. 意外的全局

未用 `var/let/const` 的赋值、隐式**泄漏到全局**（严格模式会阻断一部分）；`this` 误指向**全局**时挂载属性。

> 大体积全局缓存需**有生命周期**，用完可置 `null` 或**缩小**可见范围（模块/闭包内 + 可清理）。

### 2. 定时器、回调、观察者

- `setInterval` / `setTimeout` 长时间持有**大对象**的闭包；DOM 已移除，但闭包**仍**引用**旧**数据/节点。
- 事件**监听器**与第三方 **observer**：组件销毁时应**成对** `remove` / `unsubscribe`（现代框架多为自动，仍要**避免在回调里**闭包级联引用**整个**历史状态）。

**示例心智**：定时器里闭包了 `serverData` 与**可能已不存在**的 DOM 节点引用 → 除非 `clear` + 断引用，否则难回收。

### 3. 脱离 DOM 的**第二份**引用

把 DOM 节点放在**普通对象/Map** 里**额外**强引用，即使从文档树**摘除**，仍可达 → 需**同步**删掉**字典**里的那一份。

```js
const elements = { btn: document.getElementById('btn') }
// removeBtn 后若仍保留 elements.btn，则与「DOM 中是否还在」是两条独立问题
```

### 4. 闭包「过大」的持有

内层引用了**巨大的**、但**本可释放**的**外层**环境对象；**最小化**闭包**捕获的**变量、拆函数、在合适时机**置空**引用。

> 你不必「恐惧」闭包，但要**知边界**：谁引用了谁、何时**不可达**。

## V8 引擎（简版）

- **V8**（Chrome / Node 等）含 **Parser → Ignition 字节码 → 优化的 JIT（TurboFan 等，随版本而变）** 管线，另有 **WebAssembly** 等能力。
- **堆内存**中存放**对象/闭包**等；**GC** 在后台运行（具体调度随版本/旗标/硬件变）。

> 更深入的**编译管线**、**内联**、**去虚化**等属进阶话题；官方文档/演讲更新快，以**当前版本**为准。

## 分代与常见算法（观念）

- **分代假说**：很多对象**朝生暮死**；V8 将堆**分代**（**新生代/老生代**等，名称与**具体策略**随版本**演进**）。
- **Scavenge / Minor GC**（多针对**年轻**分代，常带**半空间复制/交换**的直觉）：频繁、**较快**、移动**仍存活**对象。
- **Mark-Sweep / Mark-Compact**（**老生代**上常见）：**标记**可达、**清理**或**整理**碎片；可能**更慢**、需要**停世界**的权衡（**增量/并发** 与具体引擎/版本强相关，勿死记一句「一定 STW 多久」）。

### 注意区分「你看到的某篇文章」

> **引用计数**在纯 RC 的教科书上示例很好讲，**环引用**是经典坑；V8 主流程**不依赖**简单 RC 作为**唯一**手段，**以官方说明为准**；面试别背错「**Major/Minor 谁管哪块**」的**一一对应**细节，答清**分代+标记+整理**的**观念**更稳。
