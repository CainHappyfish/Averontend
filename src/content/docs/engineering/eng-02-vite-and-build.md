# 构建工具与 Vite

本章衔接**本仓库技术栈、Barrel、Webpack 与 Loader 心智、Vite 的开发体验与预构建/代理/插件**等。本课**练习在本地或本仓库中完成**（如查看 `vite.config`、观察 `.vite` 缓存），不必在右侧沙箱中跑代码。

---

## 本仓库（Vite + Vue）里常碰到的点

对照你正在用的技术栈，理解这些关键词即可：

1. **`@vitejs/plugin-vue`**：让 Vite 能编译 `.vue` 单文件组件。
2. **路径 alias**（如 `@/`）：减少长相对路径，具体在 `vite.config` 里配。
3. **`import.meta.env`**：Vite 注入的环境变量，注意 **只有 `VITE_` 前缀的变量**会暴露给客户端代码（以项目配置为准）。

---

## 项目结构与 Barrel（桶文件）

**Barrel 文件**通常指某目录下的 `index.ts` / `index.js`，把多个模块再导出，让外部**一条路径**导入整组 API。

**好处**：路径短、对调用方统一入口。

**注意**：

- **导出面别过大**：能摇掉的死代码变少、易产生循环依赖。
- **循环依赖**要靠拆分模块或改引入顺序治理，必要时上工具做依赖图分析。
- 大型项目里可配合 **路径别名** 替代「什么都从 barrel 出」。

---

## Webpack 与 Loader（概念）

在 **Webpack** 体系里，**loader** 负责把非 JS 资源或特殊语法**转成**模块能处理的内容。例如 `vue-loader` 负责解析 `.vue`：把 `<template> / <script> / <style>` 交给后续 loader 与插件继续处理（模板编译、CSS 预处理、`<style scoped>` 等）。

**自定义扩展名**（如 `.md` 当模块引入）需要配置相应 `test` 与 `use: '某-loader'`，并在 `resolve.extensions` 里登记扩展名，否则无法识别。

本课程沙箱不跑 Webpack，以上只需建立**「文件 → loader → 可打包模块」**的心智；日常 Vue 3 项目里更多会接触 **Vite**。

## 为何开发时常选 Vite

和「开发时整包打一份再 HMR」相比，Vite 在**本地开发**上常见优势包括：

- **冷启动快**：先按依赖预构建，业务源码多走浏览器原生 ESM，按需请求。
- **热更新路径短**：改哪个模块就更新相关链，配合 WebSocket 推送。
- **配置相对少**：对 Vue/React 有官方或社区预设时，默认体验已经可用。

生产构建时 Vite 仍会用 Rollup 等做打包、压缩与分块，**开发体验 ≠ 生产实现细节**要分清。

---

## Vite 依赖预构建（Pre-bundling）

开发启动时，Vite 会对 **node_modules** 里需转换的部分做一次预构建，常见目的包括：

- 将 **CJS/UMD** 转为更适合浏览器加载的 **ESM** 形态。
- **合并**过于零碎的文件，减少请求次数。
- 结果缓存在 `node_modules/.vite` 等目录，**依赖锁与配置**变化时会失效重来。

这解释了「第一次启动稍慢、后面快」的一种常见原因。

---

## Vite 热更新（HMR）在做什么（简版）

1. 监听文件保存 → 2. 只编变更模块 → 3. 通过 **WebSocket** 通知浏览器 → 4. 客户端（如 `@vite/client`）按路径拉新模块并热替换 → 5. 框架层尽量**保留组件状态**。

和「整页刷新」比，HMR 减少打断心流；若改到无法热更的文件，会退化为整页刷新。

---

## 开发服务：跨域与代理

浏览器 **同源策略** 会限制不同域之间的请求。开发时常把接口代理到**与页面同源**的路径上，让浏览器以为请求发给自己的 dev server，再由**服务端**转发到真实 API。

在 Vite 中一般在 `server.proxy` 里配置；原理是**请求先到 Node 再转发**，不触发浏览器的跨域限制（与生产环境 CORS 配置是两套事）。

---

## Vite 插件与钩子（了解即可）

写插件时会在 **解析 → 读文件 → 转译 → 打包写出** 等阶段注册钩子。开发模式侧重 `configureServer`、即时 `transform`；生产构建会走完整 `build` 相关钩子。具体顺序以 [Vite 插件 API](https://cn.vitejs.dev/guide/api-plugin) 为准，无需死记。
